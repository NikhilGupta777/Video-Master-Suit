Let me read the full subtitle backend and frontend thoroughly before giving you a complete picture.

Here is a thorough audit across every layer — backend pipeline, prompts, post-processing, and frontend:
Transcription Quality

1. No audio pre-processing
The audio is sent to Gemini as-is. Downsampling to 16kHz mono WAV using ffmpeg before upload would reduce file size (often 5-10x) and actually improve transcription accuracy for speech content. Currently a 2-hour video could be a 500MB+ upload when it needs to be 50MB.

2. Seconds > 59 not caught in normalizeTs
The timestamp normalizer fixes missing hours (MM:SS → HH:MM:SS) but not invalid seconds. If Gemini produces 01:32:100,000 (seconds = 100), it passes through uncorrected. Need to carry-over: if s ≥ 60, roll over into minutes; if m ≥ 60, roll over into hours.

3. No out-of-bounds timestamp filter
After correction, there is no post-processing check that discards or clamps entries whose timestamps exceed the measured audio duration. The prompt tells Gemini to stay in bounds, but there's no enforcement in code.

4. No timestamp ordering validation
SRT entries must be in ascending order. If Gemini produces any out-of-order entries (can happen near corrections), the SRT will load incorrectly in editors. No sort/validation exists.

5. Translation entry count mismatch breaks restoreTimestamps
restoreTimestamps matches by entry number. If the translation pass accidentally adds or removes entries (splits one line into two, or merges two into one), the number map misaligns and some entries get wrong timestamps. Should compare entry counts before restoring, and flag/fallback if they differ.
Prompt Quality

6. Line length not enforced in translation
The transcription prompt says max 42 characters per line, but the translation prompt doesn't repeat this. Hindi/English translations can be 30-40% longer than Odia originals, easily blowing past line limits. Need to add the same line-length rule to the translation prompt.

7. No guidance on code-switching (English words in Odia speech)
Your videos often have words like "retirement", "minister", "documents", "scheme" spoken in English within Odia sentences. The transcription prompt doesn't address this. Should explicitly say: keep English words as-is when spoken in English, even mid-Odia sentence.

8. No instruction for handling silence/music
The transcription prompt says "if there is a quiet section, keep listening" but doesn't say "do NOT add text for music or silence." Gemini occasionally hallucinates [music] or [background noise] style entries.

9. Correction prompt could check for duplicate entries
Gemini sometimes generates two near-identical entries for the same speech. The correction pass prompt doesn't explicitly say to detect and remove them.

10. Verification prompt doesn't check entry count
The verify prompt should explicitly say "the number of entries in your output must be exactly the same as the input" — currently it can silently change the count.
Hallucination Detection (post-processing)

11. cleanupHallucinatedEntries only catches word-repetition
It catches entries where >80% of words are the same. Missing cases:

    Entries with empty text (blank subtitle)
    Entries with only punctuation (. . . or —)
    Entries with impossibly short duration (< 200ms), which are almost always artifacts
    Identical text repeated in consecutive entries

Performance & Cost

12. Audio uploaded twice
Both the transcription pass and the correction pass upload the same audio to Gemini. This is 2x the upload time. Gemini Files API lets you reuse a file by fileUri — the correction pass already does this correctly. But the file polling loop waits up to 3 minutes — if the file is already ACTIVE from pass 1, pass 2 reuses it instantly. This is actually already done right. ✅ (But worth noting the audio is in Gemini's memory for both passes.)

13. Translation + verification could be one pass
Currently: translate (pass 3) → verify (pass 4). These could be merged: "Translate this SRT and immediately proofread your own translation." This would cut the time of translation-enabled jobs by ~30-40%.
Output

14. Filename is always "subtitles.srt"
For YouTube mode, the filename should be the video title (yt-dlp can fetch it). Users have to rename the file after every download.

15. No second download for both original + translated
When translation is on, users only get the translated SRT. They have no way to also get the original-language SRT without running the job again. Should offer both as downloads.

16. No WebVTT option
Many platforms (YouTube, browser players, Premiere Pro) prefer .vtt format. SRT → VTT conversion is trivial (just change the header and , → . in timestamps).

17. No SRT validity check before delivering
Before the job is marked done, there's no final check that the output is valid SRT. A malformed output (e.g., all entries merged into one blob of text) gets silently delivered.
Frontend UX

18. No cancel button
Once a job starts, there's no way to stop it. If you paste the wrong URL, you're stuck waiting.

19. No time estimate
The "This may take 1-3 minutes" hint is static. Should be dynamic — e.g., "Audio is 11 minutes, estimated ~4 minutes remaining."

20. No copy-to-clipboard
Users often want to paste the SRT into a tool directly. There's only a download button.

21. SRT preview only shows first 8 entries
For a 230-entry file, this shows less than 4%. Should show more (20-30 entries) with proper scroll.

22. No retry on error
On failure, the only option is "Start over" which clears the URL input too. Should offer a Retry button that reruns the same job without clearing the form.

23. Step tracker shows "audio" step even for file uploads
Step 1 label is "Downloading audio from YouTube" even when the user uploads a file. Should be context-aware.

24. Translation target list is too limited
Only English and Hindi. Should include at least Odia (for cases where auto-detect gets it wrong and user wants the cleaned Odia SRT separately), and ideally other regional languages like Bengali, Telugu, Tamil.
Summary: Priority Order
Priority	Fix
🔴 Critical	Timestamp out-of-bounds filter, seconds > 59 normalization, entry count check in restoreTimestamps
🟠 High	Audio pre-processing (ffmpeg downsample), line length in translation prompt, code-switching instruction
🟡 Medium	Dual download (original + translated), dynamic filename from video title, cancel button, WebVTT output
🟢 Nice-to-have	Merged translate+verify pass, copy-to-clipboard, retry button, more translation targets
